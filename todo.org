* Agent operating rules

This file is the operational task list for the =jupiterweb= Emacs Lisp package.

The stable technical specification is =SPEC.md=. Do not edit =SPEC.md= unless the user explicitly asks for it.

The repository already contains =scrape_jupiterweb.py=. Treat that file as a reference implementation for HTTP decoding, text cleanup, discipline parsing, section extraction, instructor extraction, fallback records, and JSON-oriented field naming.

** Files to read before editing

Before changing any source file, read:

1. =SPEC.md=
2. =todo.org=
3. =DECISIONS.md=, if it exists
4. =scrape_jupiterweb.py= before working on HTTP, decoding, parsing, fallback records, or JSON export

** Work discipline

1. Work on only one task marked =NEXT= at a time.
2. Do not start a task marked only =TODO= unless instructed.
3. Move the active task from =NEXT= to =DOING= before editing files.
4. Do not implement future tasks while solving the current task.
5. Prefer small, testable changes.
6. Add or update tests whenever behavior changes.
7. Keep public APIs stable unless the task explicitly changes them.
8. Record nontrivial choices in =DECISIONS.md=.
9. Stop after completing the current task and report what changed.

** State meanings

- =TODO= :: Planned task. Do not work on it yet.
- =NEXT= :: The next task the agent should work on.
- =DOING= :: Task currently being implemented.
- =BLOCKED= :: Task cannot proceed because of missing information, failing dependency, ambiguity, or an external problem.
- =REVIEW= :: Implementation exists, but tests or manual verification are incomplete.
- =DONE= :: Task is implemented and acceptance criteria are met.
- =CANCELLED= :: Task intentionally abandoned or superseded.

** Difficulty tags

- =:D1:= :: Simple task. Low risk. Usually one file or one helper.
- =:D2:= :: Moderate task. Some integration or tests required.
- =:D3:= :: Hard task. Parser, UI, or multi-file behavior.
- =:D4:= :: Critical task. Fragile web behavior, broad integration, or release gate.

** Completion rules

A task may be marked =DONE= only if:

1. The implementation exists.
2. The task acceptance criteria are satisfied.
3. Relevant tests pass, or the task explicitly states that manual verification is enough.
4. No unrelated files were modified.
5. Important decisions were recorded in =DECISIONS.md=.

If the implementation is complete but tests were not run, mark =REVIEW=.

If the task cannot proceed, mark =BLOCKED= and explain the blocker under the task.

** Reporting template

At the end of each run, report:

- Task ID and title
- Final state: =DONE=, =REVIEW=, or =BLOCKED=
- Files changed
- Tests run
- Failures or blockers
- Next recommended task
